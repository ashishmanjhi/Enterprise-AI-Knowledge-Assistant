1. Executive Summary

The Enterprise Agentic RAG Platform is a fully local, CPU-capable Retrieval-Augmented Generation system built in Python over 15 incremental development phases. It ingests PDF and DOCX documents, indexes them in a hybrid vector+keyword store, and answers natural-language questions with provenance citations.

Beyond basic RAG, the platform adds:

  • Agentic reasoning via a LangGraph state machine that routes, rewrites, grades, generates, and verifies answers
  • Five specialised sub-agents orchestrated by a top-level router for complex research tasks
  • A knowledge graph (NetworkX) for entity/relation-enriched retrieval
  • Safety guardrails (injection, PII, toxicity, hallucination) running on every request/response
  • Multi-provider LLM support for Ollama (local), OpenAI, Anthropic, Gemini, and Azure
  • Multi-modal PDF understanding via llava:7b for chart/image descriptions

4. Core Infrastructure

4.1 Settings (backend/core/settings.py)

What it is: A single pydantic_settings.BaseSettings subclass that is the sole source of truth for every configurable parameter across all 15 phases. It loads from environment variables and .env automatically.

Design: Fields are grouped by phase with inline comments. Type annotations enforce correctness. Properties (database_url, redis_url, is_development) compose derived values. A module-level settings = Settings() singleton is imported everywhere.

Notable fields:

  • app_version = "14.0.0" — bumped at each phase
  • ollama_timeout = 300 — raised for CPU-only qwen3 inference
  • ollama_num_ctx = 4096 — prevents Ollama from defaulting to a smaller window
  • think = False — disables qwen3 internal reasoning tokens so all budget goes to the answer
  • auth_enabled = False — JWT enforcement is opt-in
  • All guardrail, agent, KG, chart, and evaluation knobs

Fixed — F-04

The startup event handler in api/main.py now validates jwt_secret_key against the insecure default at boot time. In production mode it raises a hard RuntimeError preventing startup; in development mode it logs a WARNING. db_password remains a changeme default but this only affects Postgres connectivity, not auth security.

4.2 Logging (backend/core/logging.py)

Configures a named logger (rag_platform) with both a rotating file handler (logs/app.log, 10 MB, 5 backups) and a console handler. get_logger(__name__) returns child loggers, giving full module-path context in log lines. All modules follow this convention — no bare print() calls exist in production code.

4.3 Tracing (backend/core/tracing.py)

Initialises LangSmith and OpenTelemetry at app startup. Both are opt-in via settings flags and degrade gracefully to no-ops when packages are absent or disabled. Provides trace_span(name, attrs) as an async context manager, and langsmith_callbacks() for LangGraph ainvoke calls. FastAPI HTTP instrumentation is added via add_otel_middleware(app).

4.4 Security (backend/core/security.py)

JWT helper functions: create_access_token(subject, extra_claims) and decode_access_token(token) using PyJWT HS256. Also provides generate_api_key(), hash_api_key(), and constant-time verify_api_key() for API key management. The module guards against missing PyJWT with a clear runtime error.

4.5 Auth Middleware (backend/api/middleware/auth.py)

JWTAuthMiddleware extends Starlette's BaseHTTPMiddleware. When auth_enabled=True it validates the Authorization: Bearer <token> header on every non-exempt path. Exempt paths include /health, /docs, /openapi.json, /auth/token. Decoded payload is attached to request.state.jwt_payload for downstream handlers. When auth_enabled=False (default) the middleware is a transparent pass-through.

Fixed — F-02, F-03

Auth credentials are now sourced from settings.auth_admin_password / settings.auth_user_password (loaded from .env — never committed). Password comparison uses hmac.compare_digest() for constant-time equality. Admin endpoints are protected by a require_admin() dependency guard that enforces role=admin in the JWT payload when auth_enabled=True.

5. LLM Provider Layer

5.1 Architecture

Follows the Factory + Strategy pattern. BaseLLMProvider defines the abstract interface (generate, stream, health_check, get_model_info). Six concrete providers implement it. LLMFactory.create(provider, config) selects the right class, applying configured defaults, and automatically falls back to settings.fallback_provider on failure.

5.2 LLMService vs LLMFactory

Architectural Note — Two Parallel Abstractions

The codebase has two separate LLM abstractions: LLMService (backend/llm/llm_service.py) which is a direct httpx/OpenAI client wrapper used everywhere by agents, and LLMFactory + BaseLLMProvider (backend/providers/) which is a more complete provider abstraction. In practice only LLMService is used by the RAG chain and agents. The provider layer is built but largely unused in the hot path. This is significant technical debt.

5.3 LLMService (backend/llm/llm_service.py)

The actual workhorse. Supports ollama and openai providers with sync-to-async HTTP via httpx. Key design decisions:

  • Sets think=False on Ollama to suppress internal reasoning tokens from qwen3
  • Falls back to the thinking field if response is empty (qwen3 edge case)
  • Sets num_ctx explicitly in every request to avoid Ollama's default truncation
  • Wraps timeout, connect, and HTTP errors with human-readable messages
  • Streaming uses httpx.AsyncClient.stream() and yields {"type": "token", "content": str} dicts

Weakness: Only Ollama and OpenAI are truly wired up. The generate_with_history method assumes an OpenAI-compatible client and will fail for Ollama. All the providers in backend/providers/ are unreachable from the hot path.

6. Document Ingestion Pipeline

6.1 Overview

The ingestion pipeline transforms raw uploaded files into indexed, searchable chunks. It is a five-step sequential process: Load → Chunk → Embed → FAISS Index → BM25 Index.

6.2 Execution Flow

upload_document (HTTP POST) │ ▼ IngestionPipeline.ingest_document(file_path, document_id) │ ├─1─ _load_document() │ └─ EnhancedPDFLoader.load() (pdfplumber) │ ├─ extract_text() per page │ ├─ extract_tables() → table_to_markdown() │ ├─ ChartDescriber.describe_page_images() → llava:7b │ └─ OCRProcessor (if page blank + pytesseract available) │ OR PDFLoader.load() (PyPDF2 fallback) │ OR DOCXLoader.load() │ ├─2─ _chunk_document() │ └─ DocumentChunker.chunk_pages() │ └─ RecursiveCharacterTextSplitter (1000 chars, 200 overlap) │ Tags each chunk: chunk_type ∈ {text, table, chart} │ ├─3─ _generate_embeddings() │ └─ EmbeddingService.embed_documents() │ └─ SentenceTransformer("BAAI/bge-small-en-v1.5") │ ├─4─ _index_chunks() → FAISSVectorStore.add_vectors() │ └─5─ _index_bm25() → BM25Retriever.add_documents() save() → faiss_index.bin + bm25_index.json

6.3 Document Loaders

BaseDocumentLoader is an ABC with abstract load(file_path) and supports_format(ext). It provides _validate_file(), _extract_base_metadata(), and _clean_text() utilities to all subclasses.

PDFLoader (legacy): PyPDF2-based, text-only extraction. Used as fallback when pdfplumber is absent.

EnhancedPDFLoader (Phase 13+14): pdfplumber extracts per-page text and tables simultaneously. Tables pass through table_to_markdown(). Chart images are cropped at 150 DPI and sent to llava:7b via Ollama's /api/chat endpoint. If a page yields no text/tables/charts and pytesseract is available, OCR runs as a last resort.

DOCXLoader: python-docx extracts paragraphs and tables from .docx files.

6.4 Chunking

DocumentChunker wraps LangChain's RecursiveCharacterTextSplitter. Default: 1000-char chunks, 200-char overlap, separators ["\n\n", "\n", " ", ""]. The chunk_pages() method preserves page-number metadata per chunk. Chunk indices are renumbered globally after page-level splitting.

Chunk types assigned in _chunk_document():

  • chart — content starts with [CHART]
  • table — content starts with | or [TABLE]
  • text — everything else

Weakness

The chunker applies the same 1000-char budget to table and chart chunks. Large tables may be split mid-row, breaking the Markdown structure and producing garbled context. The pdf_max_table_chunk_chars=2000 setting exists but the chunker itself doesn't honour it — it relies on pre-splitting in the loader, which is not always enforced.

6.5 ChartDescriber (backend/ingestion/chart_describer.py)

Extracts images from a pdfplumber page, filters by bounding-box area (≥5000 pts²), renders to PNG at 150 DPI, and calls ollama.chat(model="llava:7b") with a structured prompt. Returns plain-text descriptions capped at 150 words. Maximum 3 images per page. Gracefully skips on any error.

6.6 Storage

  • Raw files: data/raw/
  • FAISS index: data/vectorstore/faiss_index.bin
  • BM25 index: data/vectorstore/bm25_index.json (JSON — F-05: migrated from unsafe pickle)
  • Knowledge graph: data/knowledge_graph.json
  • PostgreSQL: 5 tables via Alembic migration (see §4.6)

Partially Addressed

A full SQLAlchemy ORM schema (backend/db/models.py) and an Alembic migration (alembic/versions/001_initial_schema.py) have been added. The five tables are: documents, document_chunks, conversations, conversation_messages, and feedback. Apply with alembic upgrade head against the running Podman Postgres. The ingestion pipeline does not yet write to these tables — that is the next integration step. Document metadata displayed in the UI still comes from filesystem stat() calls until the pipeline is updated to persist to Postgres.

7. Retrieval Layer

7.1 FAISSVectorStore (backend/retrievers/vector_store.py)

Wraps faiss.IndexFlatL2 with metadata management. Vectors are stored in the FAISS index; metadata (chunk content, document_id, filename, page_number, etc.) in a parallel Python list. Saved/loaded as a binary .bin + JSON pair. Provides add_vectors(embeddings, metadata_list, chunk_ids), search(query_embedding, top_k), delete_by_id(vector_id), delete_by_document_id(document_id), get_stats().

Vector deletion: delete_by_document_id(document_id) soft-deletes all chunks belonging to a document by replacing their metadata with {"deleted": True} and removing them from the ID→index map.

Index compaction (Phase 15): compact() rebuilds the FAISS index to physically remove all soft-deleted vectors. It calls index.reconstruct(pos) for each live entry, creates a fresh IndexFlatL2, and re-adds only the live vectors. delete_by_document_id() calls compact() automatically, so the on-disk index stays lean after every deletion.

Trade-off: IndexFlatL2 is exact search — optimal for accuracy but O(n) with corpus size. For a prototype corpus (<100K chunks) this is fine. At scale, an HNSW or IVF index would be needed.

7.2 BM25Retriever (backend/retrievers/bm25_retriever.py)

Wraps rank_bm25.BM25Okapi with configurable k1=1.5 (term saturation) and b=0.75 (length normalisation). Tokenises with lowercase split (no stemming, no stopword removal). Serialised to disk with pickle. Provides add_documents(), search(query, top_k), save(), load().

Trade-off: Pickle serialisation is fragile across Python version upgrades and poses a deserialization attack risk if the file is writable by untrusted parties.

7.3 HybridRetriever (backend/retrievers/hybrid_retriever.py)

The primary retrieval interface used by all consumers. Three modes:

  • faiss: semantic-only via FAISS
  • bm25: keyword-only via BM25
  • hybrid: both run in parallel via asyncio.gather(), then fused with Reciprocal Rank Fusion (RRF)

In hybrid mode, top_k * 2 results are fetched from each source before fusion to improve recall. An optional faiss_query parameter allows the HyDE answer to be used for FAISS while BM25 still uses the original query — a well-designed separation.

7.4 Reciprocal Rank Fusion (backend/retrievers/fusion.py)

RRF score formula: score(d) = Σ w_i / (k + rank_i(d)) where k=60 (default). Handles chunk-id deduplication across lists. Weighted variant allows FAISS and BM25 to have different influence (default 0.5/0.5). Results are sorted by fused score and capped to top_k.

7.5 Shared Instance Management

vector_store_manager.py and bm25_manager.py provide get_shared_vector_store() and get_shared_bm25_retriever() singletons. These load from disk on first call, ensuring ingestion and retrieval share the same in-memory index across request handlers.

8. Query Understanding (backend/query_understanding/)

8.1 Overview

Three LLM-powered techniques run before retrieval to improve query quality. All are opt-in per request via QueryUnderstandingOptions.

8.2 QueryReformulator

Detects vague queries (short queries, pronouns like "it/this/that", bare "how" questions without subject) and rewrites them using an LLM. Temperature 0.3, max 80 tokens. Uses the last 3 conversation turns as context for pronoun resolution. If the query is already clear, skips the LLM call.

8.3 QueryExpander

Generates N alternative phrasings (default 3) via LLM at temperature 0.7. Returns the original query as the first element. Parses numbered/bulleted lists from the LLM response. The expansions are used for multi-query retrieval (each expansion retrieved, then fused).

8.4 HyDEGenerator

Generates a short hypothetical answer (≤200 tokens) that would appear in a relevant document. This synthetic text is used as the FAISS query instead of the original question, improving semantic recall for factual and technical queries. The original question is always used for BM25.

8.5 QueryProcessor

Orchestrates all three. Runs reformulation first (serial), then expansion and HyDE in parallel via asyncio.gather(). Returns a QueryProcessingResult dataclass with all variants, timing, and flags for which techniques were applied.

Design Observation

Query understanding is only applied on the linear RAGChain path when the client explicitly passes query_understanding options in the request body. The AgentGraph and multi-agent pipeline do not use QueryProcessor — they perform their own LLM-based query rewriting inline. This creates duplicate logic and means the Phase 3 investment is only available via the plain chat endpoint.

9. Reranking (backend/rerankers/cross_encoder.py)

CrossEncoderReranker uses cross-encoder/ms-marco-MiniLM-L-6-v2 (~22 MB). A cross-encoder scores each (query, chunk) pair jointly, producing more accurate relevance scores than the bi-encoder embeddings used for initial retrieval. Batch scoring (default 16), top-N selection (default 3). Lazy model loading — downloads on first use, cached thereafter.

Controlled by settings.enable_reranking (default False) and a per-request use_reranking override. When disabled, the RAGChain instantiates self.reranker = None and skips the reranking step entirely.

10. RAG Chain (backend/llm/rag_chain.py)

The linear (non-agentic) RAG orchestrator. Used by /api/v1/chat and /api/v1/chat/stream.

Execution Flow

generate_response(query, top_k, ...) │ ├─1─ QueryProcessor.process() [optional, Phase 3] │ Reformulate → Expand → HyDE │ ├─2─ HybridRetriever.retrieve() [Phase 2] │ FAISS + BM25 parallel → RRF fusion │ faiss_query = hyde_answer (if available) │ ├─3─ CrossEncoderReranker.rerank() [optional, Phase 4] │ ├─4─ _build_prompt(query, context, history) │ System + last-5-turns history + context + question │ ├─5─ LLMService.generate(prompt) │ └─6─ _format_sources(results) Return {response, sources, metadata, query_metadata}

Key design note: The original (pre-reformulation) query is always used in the final prompt, even if HyDE/reformulation was applied for retrieval. This ensures the answer addresses the user's actual question, not the hypothetical rewrite.

11. Conversational Memory

11.1 SessionMemory (backend/memory/session_memory.py)

Per-conversation message store backed by Redis. Each conversation is a Redis list under key conv:<id>:messages (JSON-encoded entries) with a 24-hour TTL. Metadata (created_at, updated_at) is stored in a Redis hash under conv:<id>:meta. On Redis unavailability, an in-process dict fallback activates automatically — no errors surface to users.

11.2 ConversationManager (backend/memory/conversation_manager.py)

Higher-level API: record_turn(conv_id, user_msg, assistant_msg) and get_prompt_history(conv_id). Implements conversation summarisation: when message count exceeds memory_summary_threshold * 2, the oldest half is summarised by the LLM and replaced with a single system message, keeping the context window small.

Note on Lazy LLM Injection

The module-level conversation_manager = ConversationManager() singleton is instantiated with llm_service=None, disabling summarisation. The chat route never injects an LLM, so summarisation is silently disabled in practice despite the setting memory_enable_summarisation=True.

12. Guardrails & Safety (backend/guardrails/)

12.1 GuardrailsPipeline

Orchestrates all detectors in two phases:

  • Input checks (before LLM): Prompt Injection → Toxicity → PII (short-circuits on first block)
  • Output checks (after LLM): Hallucination → PII in answer

Returns a GuardrailsResult dataclass with blocked: bool, block_reason: str, checks: list[DetectionResult], and redacted_text: str | None.

12.2 Detectors

DetectorMethodPatternsDefault Block
PromptInjectionDetectorRegex (14 patterns)Jailbreak, instruction overrides, delimiter injection, role overrideYes
PIIDetectorRegex (8 patterns)SSN, credit card, email, phone, IP, API keys, DOBNo (warn+redact)
ToxicityDetectorRegex (7 patterns)Violence, self-harm, hate speech, harassmentYes
HallucinationDetectorLLM-as-judge + heuristic overlapGROUNDED vs HALLUCINATION verdictNo (warn)

Strengths: Fully offline, no API keys required, fast, clearly configurable per-type. Weaknesses: Regex-only toxicity detection misses paraphrasing, coded language, and cross-language attacks. The heuristic hallucination fallback (token overlap ratio) is a crude approximation — a response that uses different vocabulary than the context will be flagged even if semantically equivalent.

13. Agentic RAG — LangGraph (backend/agents/)

13.1 State (backend/agents/state.py)

AgentState is a TypedDict with total=False. The trace field uses Annotated[List[str], operator.add] — LangGraph's built-in list reducer that safely appends entries from multiple nodes without clobbering.

13.2 Node Functions (backend/agents/nodes.py)

Six pure async functions, each receiving (state, dependency) via functools.partial:

NodeDependencyResponsibility
route_queryllmLLM classifies query → hybrid/faiss/bm25 strategy
rewrite_queryllmRewrites vague query; increments rewrite_count
retrieveretrieverRuns HybridRetriever; serialises results to plain dicts
grade_documentsllmLLM yes/no per chunk; drops irrelevant; keeps ≥1
generatellmBuilds prompt with graded context; calls LLM
check_groundingllmGROUNDED/HALLUCINATION verdict on answer

13.3 Graph Topology (backend/agents/graph.py)

START → route_query │ ┌──────▼──────┐ rewrite_count=0 and not yet rewritten? │ rewrite? │──yes──► rewrite_query ──┐ └──────┬──────┘ │ │ no │ └──────────────┬──────────────────┘ ▼ retrieve │ grade_documents │ generate │ check_grounding │ ┌──────────▼──────────┐ │ grounded OR │ │ rewrites exhausted? │──yes──► END └──────────┬──────────┘ │ no └──► route_query (retry loop)

The graph supports up to agent_max_rewrites=2 retry loops. On each failed grounding check, the query is re-routed and potentially rewritten with a different strategy. This prevents infinite loops while still providing recovery.

13.4 Dependency Injection Pattern

Dependencies (llm, retriever) are injected via functools.partial when building the graph, not imported globally. This makes nodes unit-testable — pass mock llm/retriever and call the node directly.

14. Multi-Agent Ecosystem (backend/agents/multi/)

14.1 Architecture

Five specialised sub-agents, each implemented as its own compiled StateGraph, orchestrated by a top-level MultiAgentOrchestrator. All share a single MultiAgentState TypedDict — different agents write to different fields, so results accumulate without clobbering.

14.2 Pipeline Flow

classify_intent (LLM → research | retrieval | general) │ ┌───────▼──────────┬──────────────┐ │ research │ retrieval │ (general → retrieval path) │ ResearchAgent │ RetrievalAgent └───────┬──────────┴──────┬───────┘ └────────┬─────────┘ ▼ KnowledgeAgent (entity extraction + graph lookup) │ EvaluationAgent (quality scoring) │ GovernanceAgent (safety + attribution + confidence) │ generate_final (assemble + LLM final answer) │ END

14.3 Sub-Agent Responsibilities

AgentGraph NodesKey Logic
ResearchAgentdecompose → retrieve+answer → synthesiseDecomposes query into ≤4 sub-questions; retrieves + answers each independently; LLM synthesises findings into a summary paragraph
RetrievalAgentselect_strategy → expand_and_retrieve → rerankLLM picks strategy; generates N query expansions; parallel retrieval for all; deduplicates by chunk_id; optional reranking
KnowledgeAgentextract_entities → graph_lookup → build_contextExtracts entities from query; looks up neighbourhood in GraphStore; builds enriched context string

GovernanceAgentrun_governance_checksRuns guardrails pipeline; checks attribution completeness; checks confidence threshold; can replace answer with disclaimer

14.4 Intent Classification

The orchestrator's classify_intent node prompts the LLM with a one-word classification: research | retrieval | general. Both retrieval and general route to RetrievalAgent. Only research routes to ResearchAgent. The research path is significantly more expensive (≥ 4×N LLM calls).

Design Issue

Each sub-agent calls StateGraph.ainvoke(), which is correct for a single-agent sub-graph. However, the outer orchestrator also uses ainvoke() which runs nodes sequentially. LangGraph 0.1.19 does not natively parallelize branches of the outer graph. The research and retrieval branches are conditional (not parallel), so only one runs per query. This is fine architecturally but worth noting for future optimization.

15. Knowledge Graph (backend/knowledge_graph/)

15.1 GraphStore

Wraps a NetworkX.MultiDiGraph. Entities are nodes; relations are directed edges with predicate, confidence, and source_doc attributes. Node IDs are stable SHA-1 hashes of (entity_type, entity_text.lower()). Serialised to/from JSON via networkx.node_link_data(). A module-level get_graph_store() singleton is used everywhere.

15.2 Entity Extraction

EntityExtractor uses an LLM-first strategy: structured NER prompt requesting TYPE: entity text lines. Falls back to regex patterns (proper nouns, acronyms, version-tagged terms, quoted strings) when LLM is unavailable or fails. Configurable entity types: PERSON, ORG, PRODUCT, LOCATION, CONCEPT, TECHNICAL.

15.3 Relation Mapping

RelationMapper prompts the LLM for subject | predicate | object triples. Falls back to heuristic regex for simple X is Y, X uses Y, X has Y patterns. Confidence scores are included in the output.

15.4 Graph Retrieval

GraphRetriever performs entity substring search, then expands to the 2-hop ego-graph. HybridGraphRetriever fuses FAISS+BM25 vector results with graph-enriched results using weighted RRF (0.6 vector / 0.4 graph by default).

Scalability Concern

The entire KG is loaded into RAM as a NetworkX graph. For small corpora (<10K entities) this is fine. For large document sets, in-memory graphs become impractical. There is no pagination on entity/relation endpoints, which could produce very large responses on large graphs.

23. Code Review

Code Smells & Anti-Patterns

Duplicate LLM Abstraction

Problem: LLMService in llm/llm_service.py and BaseLLMProvider + LLMFactory in providers/ serve the same purpose. The provider layer is built but unreachable from the agent/RAG hot path. Impact: High — 6 files of dead/duplicate code; any provider feature added to providers/ doesn't reach the agents. Fix: Migrate LLMService to delegate to LLMFactory.

Fixed — F-10 Route-Level Singletons Replaced with Lazy Factories

chat.py and documents.py previously instantiated RAGChain(), IngestionPipeline(), and DocumentRetriever() at module import time. These are now @lru_cache(maxsize=1) factory functions (_get_rag_chain(), _get_ingestion_pipeline(), etc.) — lazy, mockable, and startup-crash-safe.

Fixed — F-08 Pydantic v2 @field_validator Migration Complete

api/models/chat.py and documents.py migrated from deprecated @validator to @field_validator(mode='before'). No deprecation warnings.

Fixed — F-01 Document ID Mismatch Resolved

Files saved as doc_xxxxx_original_name.pdf. GET, DELETE, and list endpoints now use prefix-glob lookup (UPLOAD_DIR.glob(f"{document_id}_*")) and strip the ID prefix before returning display filenames. Core document CRUD is fully functional.

Fixed — F-07 Conversation Manager Summarisation LLM Injected

The conversation_manager singleton is now constructed with llm_service=LLMService(). Summarisation triggers correctly when MEMORY_ENABLE_SUMMARISATION=true and the turn threshold is exceeded.

Fixed — F-06, F-06b BM25 Event Loop Blocking Resolved

Both _retrieve_hybrid() and _retrieve_bm25_only() in HybridRetriever now offload BM25Retriever.search() to the thread pool via loop.run_in_executor(None, functools.partial(...)). The event loop is no longer blocked under concurrent load.

Fixed — F-09 Dead Code Removed

backend/core/config.py (thin wrappers over settings with zero imports anywhere in the codebase) has been deleted.

No Connection Pooling / Client Reuse in LLMService

Problem: LLMService.__init__ creates one httpx.AsyncClient per instance. Multiple components each create their own LLMService(), resulting in multiple connection pools to Ollama. Impact: Low-Medium — extra memory + connections. Fix: Share a single httpx client via a module-level context or dependency injection.

Magic Numbers in Prompts

Problem: Inline token budgets (max_tokens=5, max_tokens=80, max_tokens=150) are hard-coded in node functions and agents rather than reading from settings. Impact: Low — hard to tune without code changes. Fix: Add agent-specific token settings to Settings, or at minimum define constants at module top-level.

24. Strengths

Incremental Phase Architecture

The 14-phase development approach produces a codebase where each feature is self-contained, individually documented, and independently togglable via settings flags. New engineers can understand the system by reading one phase at a time.

Comprehensive Settings Centralisation

All configurable parameters — including per-phase tuning knobs — live in a single Settings class. Environment-variable-driven with defaults. This enables zero-code configuration changes in deployment.

Graceful Degradation Throughout

Redis unavailable → in-process fallback. pdfplumber unavailable → PyPDF2. pytesseract unavailable → empty OCR result. LLM NER fails → regex fallback. llava:7b fails → empty chart description. The system never hard-crashes due to an optional dependency being missing.

Excellent Tracing Architecture

The trace field in AgentState using Annotated[List[str], operator.add] is an elegant use of LangGraph's built-in reducers to safely accumulate log entries across concurrent node executions. Every node appends its own entry — the full execution trace is always available in the response for debugging.

HyDE + Hybrid Retrieval Separation

Using HyDE output for FAISS (semantic) while keeping the original query for BM25 (keyword) is architecturally correct and thoughtful. The faiss_query parameter in HybridRetriever.retrieve() makes this separation clean without leaking HyDE into keyword search.

Dependency Injection in Agent Nodes

Using functools.partial to bind llm/retriever dependencies into node functions at graph construction time keeps nodes as pure, testable functions. This is the correct pattern for LangGraph and enables easy mocking in unit tests.

Multi-Modal PDF Understanding

The Phase 13+14 PDF extraction pipeline — text + tables (pdfplumber Markdown) + chart descriptions (llava:7b) + OCR fallback — is comprehensive. The chunk_type tagging (text, table, chart) enables downstream filtering or routing by content type.

25. Weaknesses

Persistent Metadata Store — Schema Added, Pipeline Integration Pending

SQLAlchemy ORM models (backend/db/models.py) and an Alembic migration now exist. However, the IngestionPipeline does not yet write to these tables, and document list/get endpoints still derive metadata from filesystem stat() calls. chunks_created, page_count, and author are still null in responses until the pipeline is wired to Postgres.

Agentic Paths Don't Use Phase 3 Query Understanding

The AgentGraph, RetrievalAgent, and ResearchAgent all implement their own ad-hoc query rewriting/expansion inline. The purpose-built QueryProcessor from Phase 3 is only available via the linear /api/v1/chat endpoint. This creates three separate code paths for query improvement, each with different capabilities and quality.

Agent Evaluation Is Heuristic Only

The EvaluationAgent computes scores using heuristics (source overlap ratio, answer length, confidence keywords) rather than the full RAGAS framework used in the evaluation endpoint. eval_agent_use_ragas=False by default. Scores are not comparable with the /api/v1/evaluate endpoint scores.

Feedback Data Is Append-Only JSON Lines

Feedback is stored in data/feedback.jsonl — an append-only flat file. There is no API to read, aggregate, or analyse feedback. It cannot be queried to find low-rated answers, track trends, or feed into model improvement workflows.

Fixed — Phase 15 Async Embedding via run_in_executor

Both embed_documents() and embed_query() in EmbeddingService now delegate to asyncio.get_event_loop().run_in_executor(None, partial(model.encode, ...)). The FastAPI event loop is no longer blocked during CPU-bound embedding generation. embed_batch() is covered transitively.

27. Issues & Risks

R1: Vector Index Deletion — Resolved

The DELETE /api/v1/documents/{document_id} endpoint now calls vector_store.delete_by_document_id() and bm25_retriever.delete_by_document_id(), then persists both indices to disk. Deleted documents no longer appear in future searches. The FAISS soft-delete leaves orphan float vectors in the underlying index (FAISS limitation) — a periodic rebuild is recommended for long-running deployments.

R2: In-Memory Indexes Lost on Restart

Risk: FAISS and BM25 indexes are saved to disk at ingestion time, but the singleton instances loaded from disk on startup are the only authoritative copies. If the server crashes mid-ingestion (after adding to memory but before save()), index and disk diverge. Severity: Medium.

R3: Pickle BM25 Security Risk — Resolved (F-05)

BM25 index now serialised as JSON (bm25_index.json). Legacy .pkl files are read-only for one-time migration and are never written. The pickle RCE vector is eliminated.

R4: Hardcoded Demo Credentials — Resolved (F-02)

Credentials are now loaded from settings.auth_admin_password / settings.auth_user_password (sourced from .env). Password comparison uses hmac.compare_digest(). The _DEMO_USERS hardcoded dict no longer exists.

R5: LangGraph Version Lock

Risk: The codebase pins langgraph==0.1.19, a very early version. LangGraph's API changed significantly from 0.1.x to 0.2.x and later. Upgrading is a breaking change but staying on 0.1.19 means no bug fixes or performance improvements. Severity: Medium (maintenance risk).

R6: RAGAS Compatibility

Risk: ragas==0.2.15 has specific LangChain version dependencies. The project also pins langchain==0.1.0 and langchain-core==0.1.21. Dependency conflicts during pip install are likely in fresh environments — especially on Python 3.11+ with newer transitive dependencies. Severity: Medium.

28. Recommendations

REC-1: Metadata Persistence Layer — In Progress

Done: backend/db/models.py defines ORM models for Document, DocumentChunk, Conversation, ConversationMessage, and Feedback. backend/db/session.py provides sync (Alembic) and async (FastAPI) engines and a get_db() dependency. alembic/versions/001_initial_schema.py creates all tables. Apply with alembic upgrade head.

Remaining: Wire IngestionPipeline to persist Document + DocumentChunk rows on ingestion. Update document list/get endpoints to read from DB instead of filesystem stat.

Complexity remaining: Medium.

REC-2: Vector Index Deletion — Done

Done: FAISSVectorStore.delete_by_document_id() soft-deletes all chunk vectors for a document. BM25Retriever.delete_by_document_id() filters the corpus and rebuilds the BM25Okapi model. Both are called from DELETE /api/v1/documents/{document_id} and both indices are saved to disk after deletion.

Complexity remaining: Low — periodic FAISS index compaction (remove soft-deleted float vectors) not yet implemented but not urgent.

REC-3: Consolidate LLM Abstractions

Why: Having two parallel LLM abstractions (LLMService and providers/) is confusing and means provider layer improvements don't reach the hot path.

Solution: Refactor LLMService to be a thin wrapper over LLMFactory.create(). The provider implementations already exist — just wire them in.

Complexity: Medium — requires careful testing of all endpoints after migration.

REC-4: Replace Pickle Serialisation with JSON for BM25

Why: Pickle is a security risk and fragile across Python versions. The BM25 data is plain lists of strings and dicts — easily serialisable as JSON.

Solution: Implement a JSON serialisation path in BM25Retriever.save()/load() and rebuild the BM25Okapi index from the corpus on load (same as startup does now).

Complexity: Low — the corpus is already stored as plain dicts.

REC-5: Move Route-Level Singletons to FastAPI Dependencies

Why: Module-level instantiation of RAGChain(), IngestionPipeline(), etc., fails loudly on startup if any dependency is missing, and is untestable.

Solution: Convert to @lru_cache functions exposed as Depends(get_rag_chain) in FastAPI route signatures. This enables lazy init and easy mocking in tests.

Complexity: Low — mechanical refactor.

REC-6: Add Rate Limiting

Why: Without rate limiting, a single client can exhaust Ollama resources, cause OOM, or run up cloud LLM costs.

Solution: Add slowapi (FastAPI-compatible rate limiter) with per-IP limits on /api/v1/chat, /api/v1/agent/chat, and /api/v1/multi-agent/chat.

Complexity: Low.

REC-7: Fix Conversation Summarisation

Why: Long conversations grow unbounded because the LLM is never injected into the singleton manager.

Solution: Either pass the LLM lazily in record_turn() using a get_llm_service() factory, or instantiate ConversationManager within the chat route handler where the LLM is already available.

Complexity: Low.

REC-8: Route-Level Integration Tests — Done

Done: backend/tests/test_api_routes.py adds 52 tests covering: auth (token, refresh, bad creds, role claims), document upload/list/get/delete/search/stats, chat RAG happy-path (mocked), guardrail block (400 path), streaming, direct chat, admin system-info and clear, and OpenAPI schema completeness. conftest.py provides all shared fixtures. Total suite: 203 tests, 0 failures.

REC-9: Add Retry Logic for Ollama Calls

Why: Ollama on CPU can occasionally time out on the first request while loading the model. A single retry would dramatically improve user experience.

Solution: Wrap _generate_ollama() with tenacity.retry(stop=stop_after_attempt(2), wait=wait_fixed(5)).

Complexity: Low.

REC-10: Harden Authentication for Any Deployment Beyond localhost

Why: Hardcoded demo credentials and plain-text password comparison are critical vulnerabilities.

Solution (short-term): Hash demo passwords with bcrypt in auth.py, use constant-time comparison. Solution (long-term): Integrate with an identity provider (OAuth2/LDAP/SAML) or at minimum load users from environment variables.

Complexity: Low (bcrypt) / High (IdP integration).

29. Project Assessment

Is the architecture appropriate for the project's goals?

Yes, broadly. The layered architecture — ingestion → retrieval → generation → safety — is the correct structure for an enterprise RAG platform. LangGraph is the right tool for agentic orchestration (declarative, debuggable, supports loops and conditional edges). The settings-driven, phase-tagged approach produces a system that is comprehensible and extensible.

Areas of Over-Engineering

  • Multi-agent ecosystem at v1: A 5-agent pipeline running 15+ LLM calls per query is architecturally ambitious but operationally impractical on CPU-only Ollama. The research path alone takes 4–8 minutes on CPU. This is over-engineered for the current hardware target.
  • Provider abstraction layer: Building 6 provider implementations (providers/) when only 2 are used (Ollama + OpenAI via LLMService) adds complexity without benefit until the migration from LLMService is completed.
  • Knowledge graph hybrid retriever: RRF fusion of 4 different result sources (FAISS, BM25, graph, entity lookup) adds significant complexity with diminishing returns for most practical query types.

Areas of Under-Engineering — Updated Status

  • Partial No database schema — SQLAlchemy ORM + Alembic migration added; pipeline integration pending
  • Done No vector index deletion — delete_by_document_id() implemented on FAISS + BM25 and wired into DELETE route
  • Done No API-level tests — 52 route tests added covering all major endpoints
  • Done No startup validation — JWT secret check added: RuntimeError in production, WARNING in development

What should be prioritised for refactoring?

  1. Persistent metadata store (enables correct document CRUD)
  2. Vector deletion support (correctness)
  3. LLM abstraction consolidation (maintainability)
  4. Route-level integration tests (confidence in regressions)
  5. Rate limiting (production safety)

What should remain unchanged?

  • The Settings architecture — well-designed and comprehensive
  • The logging pattern — consistent throughout
  • The guardrails pipeline — well-structured and configurable
  • The AgentGraph node/state structure — clean functional design
  • The hybrid retriever + RRF fusion implementation — correct and well-tested by the existing retrieval patterns
